# Custom Components, Dependencies, And Routing

## Custom Components

Designer-built custom components are Forms with metadata that exposes:

- Custom properties.
- Custom events.
- Optional slots.
- Optional container behavior.
- Toolbox metadata.

YAML fields:

```yaml
custom_component: true
properties:
- name: value
  type: string
  default_value: ""
  default_binding_prop: true
  description: Current value
  important: true
events:
- name: x-value-changed
  default_event: true
  parameters:
  - name: value
```

Custom property types supported by the runtime schema include:

- `string`
- `number`
- `boolean`
- `text[]`
- `enum`
- `object`
- `dict`
- `form`
- `color`
- `icon`
- `themeRole`
- `uri`
- `html`
- `recordType`
- `margin`
- `padding`
- `spacing`

In Python, custom properties are normal attributes generated by the runtime. Use property setters or event handlers to update UI when values change, depending on the component design.

Raise custom events:

```python
self.raise_event("x-value-changed", value=self.value)
```

## Custom Component Containers And Slots

A Form can expose slots to allow other Forms/components to insert children. Slot YAML:

```yaml
slots:
  default:
    target:
      type: container
      name: content_panel
    index: 0
    set_layout_properties:
      slot: default
    one_component: false
```

Fields:

- `target.type`: `container` or `slot`.
- `target.name`: Target component or target layout slot name.
- `index`: Insertion position.
- `set_layout_properties`: Layout props assigned to inserted components.
- `one_component`: Whether slot permits only one component.
- `template`: Optional default/template component.

If `custom_component_container: true`, the custom component exposes container behavior in the designer. This is special-case runtime behavior; test drag/drop and runtime addition carefully.

## HTML Templates

`HtmlTemplate` components and standard page templates use HTML with Anvil attributes.

Example template:

```html
<div class="page">
  <header anvil-name="header"></header>
  <main anvil-slot="default"></main>
  <div anvil-if-slot-empty="default">Nothing here yet</div>
  <div anvil-hide-if-slot-empty="default">Loaded</div>
</div>
```

YAML root:

```yaml
container:
  type: HtmlTemplate
  properties:
    html: '@theme:standard-page.html'
components:
- name: content_panel
  type: ColumnPanel
  properties: {}
  layout_properties:
    slot: default
```

Python:

```python
self.dom_nodes["header"].classList.add("is-compact")
```

`dom_nodes` is read-only by default but can be overridden for compatibility. Prefer using it as a lookup map, not as application state.

HTML template scripts are sanitized and loaded when the component is on page. Avoid depending on script execution order unless the template is already built that way.

## Theme Assets

Theme assets live under:

```text
theme/assets/
  standard-page.html
  theme.css
  Icons/
  images/
```

Asset URLs commonly use:

```text
_/theme/Icons/icon.svg
asset:logo.jpg
@theme:standard-page.html
```

Use existing asset conventions. If an app has custom SVG icons, check whether they are rendered as images through component `icon` properties or embedded inline in HTML.

## Theme Parameters And Roles

`theme/parameters.yaml` can define theme roles and color variables used by the designer. Runtime role behavior is CSS-class based:

```yaml
properties:
  role: [card, selected]
```

Renders as:

```css
.anvil-role-card.anvil-role-selected
```

Use semantic roles:

```yaml
role: project-card
role: [project-card, selected]
role: [action-button, danger]
```

Keep app-specific visual systems in the app. Do not promote one app's roles like `primary-button` or `learning-card-warning` into framework-wide rules unless the target app uses them.

## Dependencies

Anvil dependencies allow one app to use Forms, custom components, services, and code from another app.

`anvil.yaml`:

```yaml
dependencies:
- app_id: ZBDT7UM6GVGR7W4D
  version:
    branch: master
```

Local App Server:

```bash
anvil-app-server --app MainApp --dep-id ZBDT7UM6GVGR7W4D=SharedComponents
```

Dependency directories should usually sit next to the main app directory, not inside it.

Custom component references in form YAML often look like:

```yaml
type: form:SharedComponents.Buttons.IconButton
type: form:ZBDT7UM6GVGR7W4D:Widgets.Tabulator
```

Do not guess the reference format. Search the app for existing `type: form:` entries and follow those.

## Native JavaScript And CSS Dependencies

External scripts/styles can be inserted through `native_deps` in `anvil.yaml`:

```yaml
native_deps:
  head_html: |
    <script src="https://cdn.example.com/library.js"></script>
    <link rel="stylesheet" href="https://cdn.example.com/library.css">
```

Use this only when the app needs a browser-side library that is not already loaded. Avoid adding remote dependencies casually; they affect load time, security policy, and offline/local behavior.

## Routing

Routing is optional. Many Anvil apps use simple `open_form(...)` navigation or swap pages inside a content panel. Some apps use the Routing dependency for URL-based navigation.

Before changing routing:

- Inspect `anvil.yaml` dependencies.
- Search for `routing`, `Route`, `navigate`, `open_form`, `set_url_hash`, and startup modules.
- Identify whether the app uses hosted Anvil routing, `anvil_extras.routing`, or another package.

Common routing shape:

```python
from anvil_extras import routing


@routing.route("/", title="Home")
class Home(HomeTemplate):
    ...


@routing.route("/users/:user_id", title="User")
class UserDetail(UserDetailTemplate):
    def __init__(self, user_id, **properties):
        self.user_id = user_id
        self.init_components(**properties)
```

Other apps may use imports like:

```python
from anvil.routing import navigate
```

Treat import path and API as dependency-specific. Follow the existing app.

Good routing practice:

- Define routes centrally or in predictable page modules.
- Keep route handlers thin.
- Pass ids in URLs, not full row objects.
- Rehydrate and authorize data in server code.
- Ensure route modules are imported before launching the router.
- Preserve browser back/forward semantics where routing is already in use.

## Navigation Without Routing

Simple Anvil navigation:

```python
from anvil import open_form

open_form("Home")
open_form(Home(item=row))
```

Content-panel navigation:

```python
self.content_panel.clear()
self.content_panel.add_component(UserDetail(user_id=user_id))
```

Use the local app's established navigation pattern. Do not introduce a routing dependency just for a small change unless the user asked for it or the app already uses it.

## HTTP Endpoints

Server modules can expose HTTP endpoints:

```python
@anvil.server.http_endpoint("/webhook", methods=["POST"])
def webhook(**params):
    request = anvil.server.request
    ...
    return anvil.server.HttpResponse(200, "ok")
```

Security considerations:

- Verify signatures for webhooks.
- Avoid enabling CORS broadly.
- Use `authenticate_users` or explicit auth where appropriate.
- Do not trust `request.body_json` without validation.

## Uplink

Uplink connects external Python processes to an Anvil app:

```python
import anvil.server

anvil.server.connect("UPLINK_KEY", url="ws://localhost:3030/_/uplink")
anvil.server.wait_forever()
```

Use App Server `--uplink-key` for privileged uplinks and `--client-uplink-key` for unprivileged client uplinks.

Uplink code can register callables:

```python
@anvil.server.callable
def run_model(input_data):
    ...
```

Treat uplink keys as secrets.
